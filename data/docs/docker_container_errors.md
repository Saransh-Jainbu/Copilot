# Docker & Container Errors in CI/CD

## COPY failed: file not found / stat no such file or directory

**Root Cause**: The Dockerfile `COPY` or `ADD` instruction references a file/directory that doesn't exist in the build context. Common causes:
- The file path is relative to the build context root, not the Dockerfile location
- The file is listed in `.dockerignore`
- Typo in the filename or path
- The file is generated by a previous step that hasn't run yet

**Fix**:
1. Check `.dockerignore` — it may be excluding the file. Remove it from `.dockerignore` or restructure.
2. Verify the build context: `docker build -t myapp .` uses `.` as the context root.
3. Use `COPY ./src/ /app/src/` (explicit paths) instead of relying on relative paths.
4. For multi-stage builds, ensure the file exists in the stage you're copying from:
```dockerfile
COPY --from=builder /app/dist /app/dist
```

## Error response from daemon: manifest for X not found

**Root Cause**: Docker can't find the specified image tag in the registry. Causes:
- Typo in image name or tag
- The tag was deleted or never pushed
- Using `latest` and the image only has versioned tags
- Private registry and you're not authenticated

**Fix**:
1. Check the exact tag exists: visit Docker Hub or your registry and verify the tag.
2. Use specific version tags: `FROM node:18.19-alpine` instead of `FROM node:latest`.
3. For private registries: `docker login registry.example.com` before building.
4. For GitHub Container Registry:
```yaml
- name: Login to GHCR
  run: echo ${{ secrets.GITHUB_TOKEN }} | docker login ghcr.io -u ${{ github.actor }} --password-stdin
```

## Container exited with code 137 (OOMKilled)

**Root Cause**: The container was killed by the kernel's OOM (Out Of Memory) killer. The process inside exceeded the container's memory limit.

**Fix**:
1. Increase memory limit: `docker run -m 2g myapp` or in `docker-compose.yml`:
```yaml
services:
  app:
    deploy:
      resources:
        limits:
          memory: 2G
```
2. For Node.js: increase heap size: `NODE_OPTIONS="--max-old-space-size=4096"`
3. For Java: set JVM heap: `JAVA_OPTS="-Xmx1g -Xms512m"`
4. Profile memory usage locally to find leaks before deploying.

## port is already allocated / bind: address already in use

**Root Cause**: Another container or process is already using the same host port.

**Fix**:
1. Stop the conflicting container: `docker ps` to find it, `docker stop <id>` to stop it.
2. Use a different host port: `docker run -p 8081:80` instead of `-p 80:80`.
3. In `docker-compose.yml`, change the host port mapping.
4. In CI: use dynamic ports or ensure cleanup between test runs:
```yaml
- name: Cleanup
  run: docker-compose down -v --remove-orphans
```

## ENTRYPOINT vs CMD confusion / container starts and exits immediately

**Root Cause**: The container starts but the main process exits immediately because:
- `CMD` is overridden by docker run arguments unexpectedly
- `ENTRYPOINT` is set to a script that doesn't `exec` the main process
- The process runs in the background (daemon mode) and the container exits when PID 1 finishes

**Fix**:
1. Keep the main process in the foreground: don't use `&` or daemon mode.
2. Use `exec` in entrypoint scripts to replace the shell with the app process:
```bash
#!/bin/sh
# entrypoint.sh
exec python app.py "$@"
```
3. Use `CMD ["python", "app.py"]` (exec form) instead of `CMD python app.py` (shell form).
4. For debugging: `docker run -it myapp /bin/sh` to get a shell and check the environment.

## Docker layer caching not working in CI

**Root Cause**: CI environments typically start with a clean Docker daemon, so no cached layers exist. Every build downloads and installs everything from scratch.

**Fix**:
1. Use BuildKit caching with GitHub Actions:
```yaml
- uses: docker/build-push-action@v5
  with:
    cache-from: type=gha
    cache-to: type=gha,mode=max
```
2. Order Dockerfile instructions from least to most frequently changing:
```dockerfile
# Dependencies change less often — cache this layer
COPY requirements.txt .
RUN pip install -r requirements.txt
# Source code changes frequently — this layer invalidates often
COPY . .
```
3. Use multi-stage builds to avoid invalidating dependency layers when source changes.
